# 라우터가 아니라 오케스트레이터-위임형을 선택

고성능 CLI(월별 한도)와 저성능 CLI(일일 초기화)를 조합해 토큰을 아끼는 하네스를 만들 때,
task 전체를 난이도 분류해 CLI 하나에 배정하는 스탠드얼론 라우터 대신, 고성능 모델이
작업을 수행하면서 하위 작업만 Gemini에 위임하는 구조를 선택했다. 이유: (1) 분류 자체가
판단력이 필요한 작업이라 별도 분류기는 오분류 시 재작업 이중 지출이 발생하는 반면,
위임형은 분류를 고성능 모델의 문맥 내 판단에 맡긴다. (2) 실제 토큰 소비의 대부분은
task 단위가 아니라 하위 작업 단위(탐색·대량 읽기)에서 발생하므로 절약 효과도 위임형이
크다. 트레이드오프: 완전히 사소한 task도 오케스트레이터(고성능)를 거치므로 그 경우의
절약은 포기한다.

## Considered Options

- 스탠드얼론 라우터 (task 전체 배정) — 오분류 이중 지출, 별도 분류기 필요로 기각
- 하이브리드 (진입점 1차 분류 + 위임) — 효과 최대지만 구현 두 배, v2 후보로 보류
- 어드바이저 (추천만) — 자동화 가치 없음으로 기각
