# 속성 테스트를 2단(하드 게이트 + 경제성 판단)으로 완화하고 사용자 오버라이드를 도입

워커(Gemini CLI)의 실제 능력을 측정하지 않은 채 네 속성 전부를 하드 게이트로 두어,
보수적 추정이 이중으로 적용되고 있었다: 실패 비용은 이미 출구 규칙(재시도 1회 → 회수)이
캡핑하고 워커 쿼터는 일일 초기화라 과잉 위임의 하방이 작은데도, 게이트가 위임 시도 자체를
막아 능력을 학습할 데이터도 쌓이지 않았다. 잘못된 결과가 작업에 스며드는 것을 막는 실질
방어선은 게이트가 아니라 검증 의무(스팟체크·diff 검토)다. 따라서 속성 테스트를 2단으로
재구성한다: **하드 게이트**(속성 1 spec 완결성 — spec은 워커에게 가는 유일한 채널이라
워커 능력과 무관한 구조 제약, 속성 4 시크릿 무관 — 성능이 아닌 피해 모델 문제)는 예외
없이 유지하고, **경제성 판단**(속성 2 검증 비용 ≪ 수행 비용, 속성 3 재시도 저렴)은
금지가 아닌 저울로 강등해 애매하면 위임 쪽으로 기울인다. 사용자가 위임을 명시
지시하면(반응 발동) 경제성 판단은 오버라이드된다 — 오케스트레이터는 우려를 한 줄로 밝히되
따른다. 시크릿은 오버라이드 불가하고, spec 미완결은 거부 사유가 아니라 부족한 맥락을 물어
spec으로 굳혀 해소한다. 네거티브 리스트의 능력 추정 항목(설계 판단·핵심 로직 금지 등)은
제거하고 경제성 근거가 달린 '부적합 경향' 카탈로그로 대체한다. 완화가 맞았는지는 장부로
측정한다 — 회수를 `--reclaim`(verify=검증 재탈락 / worker=워커 불능)으로 기록해
재위임률(`retry` 필드)과 함께 판정 기준 튜닝의 근거로 쓴다.

속성 4의 존치 근거를 교체한다: 데이터 경계 논리로는 시크릿을 사내 코드와 구분할 수 없다
(경계는 enterprise 계약이 담보 — ADR 0006이 해소). 시크릿이 다른 것은 피해 모델이다 —
사본 하나하나가 즉시 행사 가능한 권한이고, 위임은 워커 요청 로그·위임 장부(spec_summary)·
대화 기록에 사본을 흩뿌린다. 사본 최소화는 계약과 무관한 보안 위생이며, 시크릿 내용을
다뤄야 하는 작업은 애초에 위임 적합한 대량·정형 작업이 아니라서 이 게이트의 비용은
사실상 0이다.

## Considered Options

- **전면 완화(네 속성 모두 판단으로 강등)** — spec 완결성은 워커가 아무리 뛰어나도 못
  넘는 구조 제약이고, 시크릿은 성능 판단이 아니다. 완화의 근거(능력 미실측)가 이 둘에는
  적용되지 않아 기각.
- **현상 유지(능력 실측 후 완화)** — 측정할 데이터 자체가 게이트에 막혀 쌓이지 않는 순환.
  완화 → 장부 관측 → 재조정이 유일한 학습 경로라 기각.
- **속성 4 삭제** — enterprise 계약 논리로는 일관되나, 사본 최소화 위생을 버리는 대가로
  얻는 게 없다(이 게이트에 막히는 위임이 사실상 없음). 기각.
- **오버라이드를 하드 게이트까지 확대** — 시크릿은 사용자 지시로도 열지 않는다(위생 규칙은
  편의와 거래하지 않음). spec 미완결은 거부가 아니라 맥락 굳히기로 해소되므로 오버라이드가
  필요 없다. 기각.
