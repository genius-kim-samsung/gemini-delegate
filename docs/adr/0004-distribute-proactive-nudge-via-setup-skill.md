# 자동 트리거는 install.py가 아니라 번들 setup 스킬이 배포한다

능동 발동(오케스트레이터가 명시 지시 없이 대량 탐색·보일러플레이트 같은 위임 적합 작업을
시작하기 직전에 스스로 위임 적합성을 검토하는 것)은 스킬 description만으로는 구조적으로
약하다 — description은 호출 판정 시점에만 참조되어 "지금 나를 중단하고 위임을 검토하라"는
자기중단을 강제하지 못한다. 그래서 호스트 메모리 파일(Claude Code `CLAUDE.md`,
Codex `AGENTS.md`)에 자동 트리거 한 줄을 둔다 — 이 파일은 매 턴 컨텍스트에 로드되어
리마인드가 강하다.

문제는 이 파일을 **스킬 배포 채널이 원칙적으로 건드리지 않는다**는 것이다. `npx skills`
(Agent Skills 공개 표준의 CLI)도, 이 저장소의 `install.py`도 스킬 디렉터리를 복사·심링크할
뿐, 사용자 소유 메모리 파일을 수정하거나 포스트-인스톨 훅을 돌리지 않는다. 즉 배포가 깔끔한
채널(스킬 파일)은 능동 발동에 약하고, 능동 발동에 강한 운반체(메모리 파일)는 표준 채널로
배포되지 않는다.

그래서 자동 트리거 배포를 **번들 setup 스킬**(`setup-gemini-delegate`)에 맡긴다. 이 스킬은
에이전트가 실행하며, 자기 설치 경로로 호스트를 감지해(`.claude` vs `.codex`, 모호하면 사용자에게
질문) 알맞은 메모리 파일을 **엄격 마커 블록**(`<!-- gemini-delegate:begin/end -->`)으로 관리한다 —
삽입·갱신(정본 재조정)·삭제. `install.py`는 스킬 복사 오프라인 안전망으로 축소하고 자동 트리거 로직은
갖지 않는다(단일 출처 = setup 스킬). 능동 발동 신뢰도는 **측정 불가**로 수용한다 — 장부는
위임한 것만 남기고 놓친 것(false negative)은 흔적을 안 남기므로, 자동 트리거는 결정론이 아니라
best-effort다.

## Considered Options

- **install.py가 `~/.claude/CLAUDE.md`를 직접 편집** — 호스트를 하드코딩해 Codex `AGENTS.md`를
  못 다루고(이 저장소의 호스트 무관 원칙 위반), 사용자가 큐레이트한 전역 설정 파일을 Python이
  조용히 편집하며, npm 채널(`npx skills`)과 이중 배포가 된다. 기각.
- **스킬 description에만 의존(자동 트리거 배포 안 함)** — 반응 발동(명시 `/gemini-delegate`·"위임")엔 충분하나
  능동 자기중단엔 약해, 설치만 한 다른 사용자에겐 사실상 안 걸린다. 기각.
- **PreToolUse hook으로 결정론적 자동 발동** — 능동 신뢰도를 실제로 올리는 유일한 기제지만 호스트
  종속적이고 대량 Grep마다 울려 알람 피로를 낳는다. 보류(측정된 miss가 쌓이면 후보).
- **번들 setup 스킬이 호스트 감지해 메모리 파일 관리 (채택)** — `npx skills` 표준 채널로 배포되고,
  에이전트가 호스트를 인지해 CLAUDE.md/AGENTS.md를 알맞게, 마커로 갱신·삭제까지 한다.
  자동 트리거 신뢰도는 측정 불가 best-effort임을 받아들인다.
