# auto 라우팅 유지 + 수행 모델 관측 우선 (즉시 flash 고정 안 함)

위임 시 `--model`을 지정하지 않으면 gemini의 auto 라우터가 모델을 스스로 고른다.
사내망 실측 결과, `"Reply with exactly: ok"` 같은 완전 trivial 위임조차 pro
(`gemini-3.1-pro-preview`, input 26,048 토큰)로 라우팅됐다 — 위임 계약·시스템 프롬프트·툴
로딩 때문에 저판단 작업도 컨텍스트가 커서 라우터가 "복잡"으로 오판한다. 즉 **저판단 위임에
귀한 pro 쿼터가 새는 것이 측정된 사실**이다.

그럼에도 즉시 flash 고정으로 가지 않고, **auto를 유지한 채 수행 모델을 장부 + 대면 한 줄로
관측**하기로 했다. 이유: (1) 모델 id가 계속 바뀐다(오늘 pro가 `gemini-3.1-pro-preview`) —
고정 id를 하드코딩하면 곧 낡는다. (2) 관측 데이터 없이 flash로 고정하면, flash로는 품질이
무너지는 위임 유형을 미리 걸러낼 수 없다. (3) 리포트가 쌓는 장부가 전환 시점과 대상을
데이터로 정당화한다. 전환은 read 위임 기본 모델을 상수 한 곳(`DEFAULT_READ_MODEL`)에서
flash id로 바꾸는 **한 줄 변경**으로 열어둔다. 트레이드오프: 관측 기간 동안 pro 쿼터 낭비를
감수한다.

수행 모델 추출은 gemini 전용이다 — agy는 `--output-format`이 없고 모델명 체계도
멀티프로바이더 표시명(`Gemini 3.5 Flash (High)`, `Claude Opus 4.6` 등)이라 이질적이라,
백엔드별로 분기해 agy는 '미지원'으로 기록한다(dev 전용 탈출구이므로 투자하지 않는다).

## Considered Options

- **즉시 flash 고정** — 저판단 위임에 최적이고 pro를 즉시 아끼지만, 관측 없이 고정하면 품질
  회귀 위험 + 변동하는 모델 id 하드코딩. 보류(전환 후보).
- **auto 그대로 + 관측도 안 함 (현행)** — 어떤 모델이 실제로 돌았는지 장부에 안 남아 튜닝
  근거가 없고, pro 낭비가 비가시. 기각.
- **auto 유지 + 수행 모델 관측 (채택)** — 절약은 미뤄지지만 전환을 데이터로 정당화할 수 있고
  되돌리기 쉽다.

## 결정 갱신 (2026-07-14): read 기본을 flash로 고정

위에서 열어둔 전환 조건("전환은 데이터로 정당화")을 충족해, read 위임 기본 모델을
`gemini-3.5-flash`로 고정한다(`DEFAULT_READ_MODEL`, gemini 백엔드 전용). 근거: 공개 벤치·실측상
3.5 Flash가 코딩·에이전틱(=위임 대상 작업의 성격)에서 3.1 Pro 동급 이상이면서 더 싸고 빠르다.
위임 대상은 4속성 필터로 저판단이 보장되고 검증+재시도가 품질 회귀를 흡수하므로 flash 고정
리스크가 낮다. pro가 우위인 하드 리즈닝·신규 연역은 판단이 필요해 애초에 위임 대상이 아니다.

write 위임은 auto를 유지한다(파일 수정은 회귀 비용이 커 관측을 이어간다). 수행 모델 관측 라인은
그대로 둬 flash가 실제로 돌았는지(쿼터로 다른 모델로 밀렸는지)를 계속 확인한다. 되돌리려면
`DEFAULT_READ_MODEL`을 지우거나 pro id로 바꾸면 된다.
