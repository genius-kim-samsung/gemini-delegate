# Gemini 위임 (gemini-delegate)

너(오케스트레이터)는 작업 전체를 책임진다. 위임은 하위 작업 하나를 워커(Gemini CLI)에
넘기고 결과만 회수하는 것이며, task 전체를 넘기는 라우팅이 아니다.

## 1. 위임 적합성 판정 — 속성 테스트

난이도가 아니라 속성으로 판정한다. 속성은 두 층이다 — **하드 게이트**(1·4)는 워커 성능과
무관한 제약이라 예외가 없고, **경제성 판단**(2·3)은 금지가 아니라 저울이다. 실패 비용은
출구 규칙이 캡핑하고(재시도 1회 → 회수) 워커 쿼터는 일일 초기화라 과잉 위임의 하방이
작다 — **애매하면 위임하는 쪽으로 기울여라**(배경 `docs/adr/0008`).

### 하드 게이트 — 걸리면 위임 불가

- **속성 1. Spec 완결성** — 대화 맥락 없이 자체 완결된 지시문을 한 번에 쓸 수 있다.
  spec은 워커에게 가는 유일한 채널이라, 워커가 아무리 뛰어나도 spec에 없는 맥락은 못 본다.
  "적당히", "지금까지 논의한 대로"가 필요하면 부적합 — 단, 이는 거부 사유가 아니라 신호다:
  맥락을 spec에 굳혀 쓸 수 있으면 굳혀서 위임하라.
- **속성 4. 시크릿 무관** — spec과 대상 경로에 시크릿(자격증명·비밀키·토큰류)이 포함되지
  않는다. 데이터 경계 문제가 아니다 — 일반 사내 코드·문서·로그는 배포 전제(사내 enterprise
  계약)가 담보한다(`docs/adr/0006`). 시크릿이 다른 것은 사본 하나하나가 살아있는 권한이라는
  점이다: 위임하면 워커 요청 로그·위임 장부·대화 기록에 사본이 흩어지므로 사본 최소화
  원칙상 넣지 않는다(`docs/adr/0008`). 대상 트리에 시크릿 파일(`.env`, 키·인증서 등)이
  섞여 있으면 [제약]으로 해당 경로를 제외하고 위임하라.

### 경제성 판단 — 저울질하되, 애매하면 위임

- **속성 2. 검증 비용 ≪ 수행 비용** — 결과 확인이 직접 수행보다 훨씬 싼가. 검증하려면
  결국 사고를 다시 해야 하는 작업(설계 판단, 트레이드오프 분석)은 위임해도 아끼는 게 없다.
- **속성 3. 재시도 저렴** — 실패해도 다시 시키거나 회수하는 비용이 낮은가. 부분 실패가
  다른 작업을 오염시키면 비싸지고, 테스트 같은 기계적 검증 수단이 있으면 싸진다.

### 사용자가 위임을 지시하면 — 경제성 판단 오버라이드

사용자의 명시 위임 지시(반응 발동)는 속성 2·3 판정을 대체한다. 우려가 있으면 한 줄로
밝히되 따르라. 하드 게이트만 남는다: 시크릿(4)이 걸리면 이유를 말하고 거부하라.
spec 완결성(1)이 부족하면 거부하지 말고 부족한 맥락을 물어 spec으로 굳힌 뒤 위임하라.

### 카탈로그 (예시)

| 적합 | 부적합 경향 — 이유 |
|---|---|
| 수십 개 파일 탐색·구조 요약 | 모듈 인터페이스 설계·트레이드오프 분석 (검증 = 사고를 다시 하기) |
| 대량 로그/빌드 출력에서 오류 패턴 추출 | 버그 원인 판정 (증거 수집까지만 위임, 판정은 직접) |
| 정형 보일러플레이트 생성 (테스트 골격, DTO 등) | 커밋 메시지 (맥락 필요, 직접 쓰는 게 더 쌈) |
| 기계적 반복 수정 (rename, import 정리, 포맷 통일) | 검증 수단 없는 핵심 로직 수정 (실패가 비쌈 — 테스트가 커버하면 적합 쪽) |
| 문서 초안·번역·표 변환 | |
| 테스트가 커버하는 로직 구현 (검증 = 테스트 실행) | |

## 2. Spec 작성 템플릿

위임 spec은 자체 완결이어야 한다. 다음 구조로 작성하라:

```
[목표] 무엇을 왜 하는지 한 문단
[대상] 구체적 경로/파일 패턴 (워커가 직접 읽는다 — 내용을 붙여넣지 마라)
[출력] 기대하는 출력 형식과 분량
[제약] 하지 말 것, 경계 조건
```

근거 인용·수정 범위 제한 같은 위임 계약은 delegate.py가 자동 첨부하므로 spec에 반복하지 마라.

## 3. 실행

이 스킬 디렉토리의 delegate.py를 사용하라 (`<skill-dir>`은 이 SKILL.md가 있는 디렉토리):

```bash
# 읽기 위임 (탐색·요약·분석) — 읽기 전용이 플래그로 강제됨
python "<skill-dir>/delegate.py" --type read "spec..."

# 기계적 쓰기 위임 — 파일 수정만 자동 승인됨 (임의 셸 실행 불가)
python "<skill-dir>/delegate.py" --type write --spec-file spec.md

# spec이 길면 --spec-file 또는 stdin을 써라 (인용 문제 회피)
```

- 결과는 stdout으로 반환된다. 위임은 blocking이다.
- 기본 타임아웃 600초, `--timeout`으로 조정. 모델은 `--model`로 오버라이드.
- gemini 위임은 실제 **수행 모델**(auto 라우터가 고른 결과)을 stderr에
  `[delegate] 수행 모델: …`로 보고한다. **이 줄을 사용자에게 함께 전달하라**
  (사용자가 어떤 작업이 pro/flash 중 무엇으로 돌았는지 인지하도록). agy는 구조화 출력이
  없어 '미지원'으로 표기된다.
- 모든 위임은 `~/.claude/gemini-delegate/ledger.jsonl`에 자동 기록된다 (`model` 필드 = 수행 모델).

## 4. 결과 검증 — 생략 불가

- **읽기 위임**: 이후 작업의 전제가 되는 하중 큰 주장을 골라, 인용된 `경로:줄번호`를
  직접 읽어 스팟체크하라. 근거 인용이 없는 주장은 검증 불가로 취급하고 사용하지 마라.
- **쓰기 위임**: `git diff`를 반드시 검토하라 (git 저장소가 아니면 워커가 보고한
  수정 파일을 직접 읽어 확인하라). 검토 전에는 완료로 취급하지 마라.

## 5. 실패 처리 — 출구 규칙

- **검증 탈락 1회차**: 무엇이 틀렸는지 피드백을 spec에 담아 `--retry`로 정확히 1회 재위임.
- **재탈락**: 회수하라 — 네가 직접 수행한다. 추가 재시도 금지.
  `--reclaim verify "spec 요약"`으로 장부에 남겨라.
- **워커 불능** (쿼터 소진·타임아웃·실행 오류): 재시도 없이 즉시 회수하고,
  사용자에게 한 줄로 고지하라 (예: "Gemini 쿼터 소진으로 직접 수행합니다").
  `--reclaim worker "spec 요약"`으로 장부에 남겨라.

회수 기록은 완화된 판정 기준(속성 2·3)이 실제로 어디서 실패하는지 배우는 근거다 —
재위임률(`retry` 필드)과 함께 위임 기준 튜닝에 쓰인다.
